# deep-extract 추출 설계 — 브라우저 단독 처리 (B안 확정)

> 작성일: 2026-06-05 · 대상 커밋: `7ad4b61`(UI 전면 개편) 기준 워킹트리
> 개정 이력: 서버 전환안 검토(2026-06-04) → **비용·복잡도·프라이버시 상실 때문에 철회** → **브라우저 단독 + 입력 한도 축소(B안)로 확정**(2026-06-05).
> 목표: **브라우저 단독 처리**(파일 서버 전송 없음 = USP 유지), 범용 포맷 **재귀 해제**, **OOM/손상 없이** 단순한 코드로.

---

## 0. 요약 (TL;DR)

- **처리 위치: 브라우저 단독.** 서버 없음 → 비용 0, "파일 서버 전송 없음" USP 유지, 서버 보안/운영 과제 자체가 없음.
- **입력 한도(업로드 기준): 최대 20개 / 파일당 1GB / 총 2GB.** 실사용(MB 단위 다수 파일)의 1000배라 충분, 동시에 모든 포맷이 WASM 메모리 안에서 안전.
- **지원 포맷: zip, rar, 7z, tar, gz, bz2, zst** (+ `tar.gz/tar.bz2/tar.zst` 조합). **`libarchive.js` 단일 엔진**으로 전부 처리(분기 없음).
- **재귀 깊이: `maxDepth = 5`** (이중 확장자 `.tar.gz`가 2단계 차지 감안). 폭발 방어 본진은 **누적 출력 캡 + 엔트리 압축비 가드**.
- **출력: `client-zip`(ZIP64 스트리밍) → OPFS 파일 싱크 → 다운로드.** fflate(ZIP64 미지원)·FSA(불필요) 제거.
- **메모리:** libarchive `extract()`는 **엔트리를 통째로 메모리에 적재**(진짜 바이트 스트리밍 아님). 피크 ≈ **허용된 가장 큰 엔트리 1개** → **추출 전 엔트리 크기·압축비 게이트**로 제한(§3.4). 출력 ZIP만 OPFS로 스트리밍.
- **출력/폭주 방어:** 추출 전 엔트리 게이트(OOM 1차) + **누적 출력 캡 20GB(백스톱)** + 시작 전 `storage.estimate()` 점검.

## 1. 결정 배경

### 1.1 왜 서버로 안 가는가
서버 전환을 검토했으나(2GB/5GB 대용량 처리 목적), 다음 이유로 철회:
- **비용:** 1인 개발에 서버/스토리지/이그레스 비용 부담.
- **복잡도:** 업로드 스토리지, 작업 큐, Zip Slip 격리, 테넌트 격리, 레이트리밋, 디스크 용량 산정, TTL 정리 등 **부수 과제 폭증**.
- **프라이버시:** 기존 USP "파일 서버 전송 없음"을 버리게 됨.

### 1.2 왜 한도를 축소하는가
- **실제 사용 사례 = "압축 파일 10개를 일일이 풀기 귀찮아서" (MB 단위).** 대용량(수 GB)은 사용자가 직접 풀지, 이 도구를 쓸 일이 드묾.
- 한도를 **1GB/파일·2GB 총량**으로 낮추면, **모든 포맷(7z/rar 포함)이 libarchive WASM의 4GB 벽 한참 아래**라 브라우저에서 안전하게 처리됨.
- 동시에, 대용량을 욱여넣으려던 **OPFS 다단 스테이징 복잡도가 통째로 사라짐.**

> **포맷 vs 무제한은 양립 불가였음:** ZIP/TAR은 스트리밍으로 수 GB 가능하나, **7z/rar는 WASM(libarchive/unrar)에 의존**해 통째로 메모리에 적재 → 수 GB 불가. "범용 포맷"을 택했으므로 "무제한"은 포기하고 **한도 축소**로 7z/rar까지 안전하게 지원한다. (향후 'ZIP 무제한' 수요가 보이면 zip.js 스트리밍 경로를 보너스로 덧붙일 수 있음 — 현 범위 밖.)

## 3. 아키텍처

### 3.1 입력·추출 — libarchive 단일 엔진, 워커, 재귀

- **엔진:** `libarchive.js`(WASM) **단일**. zip/rar/7z/tar/gz/bz2를 한 엔진으로 처리 → **포맷 분기 없음**(fflate zip 경로 불필요, 한도가 낮아 libarchive로 충분).
- **엔트리 단위 추출:** `extractFiles()`(아카이브 전체 동시 적재) 대신 **`getFilesArray()`로 헤더 목록 → 엔트리별 `extract()`**. 단, libarchive의 `extract()`는 해당 엔트리를 **통째로 ArrayBuffer/File로 적재**하므로 *진짜 바이트 스트리밍이 아니다* — 피크 힙 ≈ **현재 엔트리 1개의 해제 크기**. 따라서 거대 단일 멤버 1개로도 OOM 가능 → **추출 *전*에 엔트리 크기/압축비를 게이트**(§3.4)로 막는 것이 필수.
- **재귀:** 추출된 엔트리가 다시 아카이브면(`isArchive`) `maxDepth=5`까지 재귀. 내부 아카이브는 `File`/`Blob`로 libarchive에 재투입.
- **워커:** libarchive는 워커에서 동작(기존 구조 유지). 메인 스레드는 UI/진행률만.

### 3.2 출력 — client-zip(ZIP64) → OPFS 싱크 → 다운로드

```
extract(엔트리) ──► client-zip 스트림(ZIP64, store-only) ──► OPFS 파일에 스트리밍 기록
                                                              완료 후 OPFS 파일 → 다운로드(object URL)
```

- **`client-zip`:** ZIP64 기본 지원 → 출력이 4GiB·6.5만 엔트리를 넘어도 **안 깨짐**. store-only(재압축 안 함).
- **OPFS 단일 싱크:** 출력 ZIP을 OPFS 파일 1개에 **스트리밍 기록**. 모든 모던 브라우저(크롬/파폭/사파리) 지원. **FSA는 사용하지 않음**(필수 아님, 분기 제거로 단순화).
- **다운로드:** 완료된 OPFS 파일을 `File`로 읽어 object URL로 다운로드. 디스크 기반이라 다운로드 시에도 메모리 안 터짐.

> ⚠️ OPFS는 **출력 ZIP 1개를 흘려보내는 단일 싱크**로만 쓴다. 과거 복잡했던 입력 다단 스테이징과는 용법이 완전히 다름(훨씬 단순).

### 3.3 메모리·용량 설계 (핵심)

| 우려 | 설계 |
|---|---|
| 출력 4GB↑ 시 ZIP 손상? | **client-zip ZIP64** → 크기/엔트리 수 무관하게 정상 |
| 출력 크면 메모리 터짐? | 출력 ZIP은 **OPFS(디스크)로 스트리밍** → 출력 크기 ≠ 메모리. 단 **입력 엔트리는 `extract()`가 통째로 적재** → 피크 ≈ 허용된 최대 엔트리 1개 → **추출 전 엔트리 크기 게이트**로 제한(§3.4) |
| 거대 단일 멤버로 OOM? | `extract()` 전에 **선언된 해제 크기**로 거부(§3.4) — 적재되기 *전에* 막음 |
| 무한 팽창(zip-bomb)? | **추출 전 압축비 게이트 + 누적 출력 캡 20GB 백스톱** → 친절히 중단(§3.4) |

> 브라우저 처리라 **zip-bomb이 터져도 피해는 사용자 자기 디스크**(서버 없음). 캡의 목적은 서버 방어가 아니라 **선량한 사용자의 디스크 보호 + 깔끔한 중단 안내**.

### 3.4 안전 가드

- **엔트리 사전 게이트 (OOM 1차 방어, 추출 *전*):** `getFilesArray()` 헤더의 **선언된 해제 크기**를 보고, 엔트리별 **메모리 예산(조정 가능, 예: 수백 MB~1GB)** 초과 시 `extract()` 호출 **전에** 스킵+보고. **크기 미상**이거나 **압축비 비정상(예: 100배 초과)** 인 엔트리도 추출 전 거부. → `extract()`가 통째로 적재하기 *전에* 막아야 OOM을 실제로 방지(추출 후 누적 집계로는 이미 늦음).
- **누적 출력 캡: 20GB (2차 백스톱).** 입력 2GB의 ~10배 = 정당한 팽창을 모두 덮음(실사용 수십 MB). 누적 초과 시 명시적 중단.
- **사전 디스크 점검:** 시작 전 `navigator.storage.estimate()`로 OPFS 여유 확인 → 부족하면 **시작 전에** "공간 부족" 안내(중간에 터지는 것 방지). 가능하면 `navigator.storage.persist()` 요청.

### 3.5 실패 시맨틱 (정확성)

- 파일별 **성공/실패/스킵 카운트 집계** → 콜백/상태로 전파.
- 실패가 1건이라도 있으면 **성공 토스트 대신 부분 성공 경고**("20개 중 18개 추출, 2개 실패").
- **산출물이 사실상 비는 치명적 실패**(최상위 열기 실패 등)는 성공 경로 차단.
- 미지원/암호 걸린 중첩 아카이브 → 추출하지 않고 **일반 파일로 보존 + 스킵 보고**(성공 둔갑 금지).

### 3.6 경로 안전 (flatten·preserve 공통, 패커 쓰기 *전*)

추출된 엔트리명은 **출력 ZIP에 쓰기 전에** 반드시 정규화·검증한다. 브라우저 내부 처리 자체는 샌드박스라 안전하지만, **사용자가 받은 ZIP을 로컬에서 풀 때** Zip Slip/덮어쓰기 위험이 되므로 다운스트림 안전을 위해 필요.
- `../`·절대경로·드라이브 경로(`C:\`)·UNC·예약어(CON, NUL 등)·제어문자 → **거부 또는 안전 치환**.
- **심볼릭/하드링크 엔트리** → 따라가지 않음(거부/스킵+보고).
- 정규화 후 **이름 충돌** → dedupe 규칙(예: `name (1).ext`).
- **flatten·preserve 양쪽 모드 모두**에 동일 적용. preserve가 나중 단계에 켜지더라도, 이 안전 계층은 **Phase 0부터 패커가 공유**한다.

### 3.7 정리·생명주기

- OPFS 출력/임시 핸들을 **생성 즉시 정리 핸들 확보** → 성공/실패/중단 **모든 경로 `finally`에서 제거**(정상 종료 시 1차 정리).
- **시작 시 sweep가 권위 있는 복구 경로(authoritative).** 앱 시작 때 고아 세션(`de-*`/출력) 디렉터리를 탐지·제거 → **강제 새로고침·탭/브라우저 강제 종료**로 정리가 끝까지 못 돌아도 **다음 실행에서 반드시 회수**.
- **언로드 정리는 best-effort일 뿐.** `pagehide`/`beforeunload`에서 AbortController 중단 + OPFS 삭제를 *시도*하지만, **브라우저가 비동기 OPFS 삭제 완료 전에 페이지를 종료할 수 있어** 보장으로 삼지 않는다(보장은 위의 시작 sweep).
- **UI 취소 버튼 제공.** 추출 중 취소하면 **AbortController로 진행 중 작업을 중단**하고 위 1차 정리(`finally`)를 실행 — 페이지가 살아있으므로 **정리가 확실히 완료**된다. 같은 AbortController를 `pagehide`/`beforeunload`에도 연결해 페이지 이탈 시에도 중단·정리를 *시도*하되, 그 경로는 best-effort(보장은 시작 sweep).

## 4. 무엇이 해결되는가 (기존 결함 대비)

| 기존 결함 | B안에서 |
|---|---|
| fflate ZIP64 미지원 → 출력 손상 | **client-zip(ZIP64)** 로 근본 제거 |
| `extractFiles()` 전체 적재 OOM | **엔트리 단위 추출 + 추출 *전* 크기/압축비 게이트**(extract()는 엔트리를 통째로 적재하므로 *사전* 게이트가 핵심) |
| 거대 단일 멤버 / 경로 탈출(Zip Slip) | **추출 전 크기 게이트 + 패커 쓰기 전 경로 정규화**(§3.4·§3.6) |
| WASM 4GB 벽 | **입력 한도 2GB로 축소** → 벽 한참 아래, 모든 포맷 안전 |
| OPFS 쿼터·다단 스테이징 복잡도 | **출력 단일 싱크로 단순화** + 사전 estimate + finally 정리 |
| 실패가 성공으로 둔갑 | **성공/실패/스킵 집계 + 부분성공 경고** |

## 5. 실행 계획

우선순위: **정확성 → 메모리/안정성 → 정리/UX.**

### Phase 0 — 정확성 & 핵심 경로 (반나절~1일)
- [ ] 출력 패커 **client-zip(ZIP64)** 확정, fflate 제거.
- [ ] libarchive **엔트리 단위 추출** + 재귀(`maxDepth=5`).
- [ ] **엔트리 사전 게이트(필수 불변식):** 추출 *전* 선언된 해제 크기·압축비로 메모리 예산 초과/크기 미상 엔트리 거부 → OOM 1차 방어(§3.4).
- [ ] **경로 안전(필수 불변식):** 엔트리명 정규화·충돌·심링크/절대경로/예약어 거부 — flatten·preserve 공통, **패커 쓰기 전** 적용(§3.6).
- [ ] 출력 **OPFS 단일 싱크** + 다운로드.
- [ ] **실패 시맨틱**(성공/실패/스킵 집계, 부분성공 경고).

### Phase 1 — 메모리·용량·정리 가드 (1일)
- [ ] **누적 출력 캡 20GB(2차 백스톱).** (엔트리 사전 게이트는 Phase 0)
- [ ] 시작 전 **`storage.estimate()` 점검** + `persist()` 요청.
- [ ] **취소 버튼 + OPFS 정리:** UI 취소 버튼 → AbortController 중단 → `finally` 제거(1차). **시작 sweep를 권위 복구 경로로**(언로드 정리는 best-effort §3.7).

> ### Phase 2 — UI 확장성 (보류) ⏸
> **메인 기능 완성 후 진행.** (파일 목록 가상화/상한 렌더, dispatch 배치 플러시, 로그 링 버퍼 등.) — 본 작업 범위 밖.

### Phase 3 — 마감 (메인 기능 안정화 후)
- [ ] 바이트 기반 진행률 정교화.
- [ ] `preserve` 모드 활성화(경로 안전 계층은 Phase 0에 이미 존재 §3.6).
- [ ] (선택) 'ZIP 무제한' 스트리밍 경로(zip.js) 보너스 추가 검토.

| 총 2GB 혼합 업로드 | 한도 경계 | 정상 처리 |
| 출력 5GB 이상(압축 잘 풀리는 입력) | **출력 메모리/ZIP64** | 메모리 상수 유지, ZIP **정상 열림**(손상 X) |
| 5단계 중첩 + zip-bomb | 재귀·가드 | 깊이 초과 스킵, 추출 전 압축비/20GB 캡에서 안전 중단 |
| **단일 거대 멤버(선언 크기 큼) 아카이브** | **엔트리 사전 게이트** | `extract()` 호출 전 거부, OOM 없음 |
| **`../`·절대경로·심링크 포함 아카이브** | **경로 안전** | 정규화/거부, 위험 경로가 출력 ZIP에 안 들어감 |
| 7만 개 소형 파일 | 엔트리 수 | 정상 ZIP(6.5만 한계 없음) |
| 미지원/암호 중첩 아카이브 | 실패 시맨틱 | 스킵+보고, 성공 둔갑 X |
| 정상+손상 혼합 | 부분 성공 집계 | "N개 중 M개 성공" 정확 |
| 처리 중도 실패(예외) | finally 정리 | OPFS 출력/세션 즉시 정리됨 |
| **사용자 취소(버튼)** | **AbortController 중단** | 즉시 중단 + OPFS 정리 완료(페이지 생존) |
| **강제 새로고침/탭·브라우저 강제 종료** | **시작 sweep 권위 복구** | **다음 앱 시작 시** 고아 세션 탐지·제거(언로드 정리에 의존 X) |
| OPFS 공간 부족 | 사전 점검 | 시작 전 "공간 부족" 안내(중간 크래시 X) |

- **메모리 측정:** DevTools로 피크 힙이 **총량이 아니라 허용된 최대 엔트리 1개 크기 수준**에 수렴하는지(사전 게이트 작동) 확인.

## 7. 변경 요약

| 영역 | 이전 | B안 | 이유 |
|---|---|---|---|
| 처리 위치 | (서버 전환안) | **브라우저 단독** | 비용 0, USP 유지, 복잡도↓ |
| 입력 한도 | 2GB/5GB | **1GB/2GB(업로드)** | 실사용 충분 + 7z/rar까지 안전 |
| 추출 엔진 | libarchive `extractFiles()` | libarchive 엔트리 단위 + **추출 전 크기 게이트** | 피크 ≈ 최대 엔트리 1개(OOM 방어) |
| 엔진 수 | libarchive + fflate | **libarchive 단일** | 분기 제거, 단순화 |
| ZIP 출력 | fflate(ZIP64 X) | **client-zip(ZIP64)** | 손상 근본 제거 |
| 출력 싱크 | (다양) | **OPFS 단일**(FSA 제거) | 보편 지원 + 단순 |
| 재귀 깊이 | 12 | **5** | 정상 케이스 충분 |
| 출력 용량 | (미정의) | **출력 OPFS 스트리밍 + 추출 전 엔트리 게이트 + 20GB 백스톱** | 메모리·폭주 차단 |
| 경로 안전 | (미정의) | **패커 쓰기 전 정규화·충돌·심링크 거부**(Phase 0) | 다운스트림 Zip Slip 차단 |

> 핵심: **한도를 실사용 규모로 낮추니, 브라우저 단독으로 범용 포맷(zip/rar/7z/tar) 재귀 해제가 단순하게 성립**한다. 손상은 client-zip(ZIP64), OOM은 **추출 전 엔트리 크기/압축비 게이트**(extract()가 통째로 적재하므로 *사전* 차단이 핵심), 출력 폭주는 OPFS 싱크 + 20GB 백스톱, 경로 위험은 **패커 쓰기 전 정규화**로 막는다. 잔존 정리는 **시작 sweep가 권위 복구**(언로드 정리는 best-effort). 서버의 비용·보안·운영 과제는 애초에 발생하지 않으며, "파일 서버 전송 없음" USP를 지킨다. UI 확장성(Phase 2)은 메인 기능 완성 후로 보류.
